# Запити зацікавлених осіб

# Вступ

У нашій лабораторній роботі ми розглянемо тему "Система управління відкритими даними".

Наш очікуваний результат - отримати розуміння про систему управління відкритими даними та навчитися використовувати її для збору та публікації.

# Мета

Створитити систему управління відкритими даними, яка забезпечить всі необхідні аспекти використання інформації та буде відповідати вимогам розповсюдження даних.

# Контекст

Система управління відкритими даними - це інструмент, який дозволяє збирати, обробляти та надавати доступ до відкритих даних. Вона може бути корисною для розробки нових продуктів та послуг, поліпшення рішень та прийняття різних стратегічних рішень.

# Посилання

# Характеристика ділових процесів

# ID: sign_in_user

# НАЗВА: Вхід користувача в систему

# УЧАСНИКИ:

- Користувачі
- Система

# ПЕРЕДУМОВИ:

- Користувач має аккаунт в системі та правильні дані для входу

# РЕЗУЛЬТАТ:

- Користувач має доступ до функціоналу системи

# ВИКЛЮЧНІ СИТУАЦІЇ:

- Неправильний логін або пароль 
- Відсутність аккаунту в системі

# ОСНОВНИЙ СЦЕНАРІЙ:

1. Користувач вводить свій логін та пароль у форму входу в систему.
2. Система перевіряє введені дані та шукає користувача в базі даних.
3. Якщо введені дані вірні та користувач знайдений в системі, система надає доступ до функціоналу системи для користувача.
4. Якщо введені дані невірні або користувач не знайдений в системі, система повідомляє користувача про помилку та просить повторно ввести правильні дані.

# ID: add_data

# НАЗВА: Додавання даних до системи (Адмін)

# УЧАСНИКИ:

- Адмін

# ПЕРЕДУМОВИ:

- Система зберігає дані користувачів та дозволяє їх редагування адміністраторам.

# РЕЗУЛЬТАТ:

- Дані користувачів збережені та доступні для подальшого використання.

# ВИКЛЮЧНІ СИТУАЦІЇ:

- Недостатньо прав доступу або введені некоректні дані користувача.

# ОСНОВНИЙ СЦЕНАРІЙ:

1. Адмін входить до системи зі своїми обліковими даними.
2. Адмін обирає опцію додавання даних.
3. Система запитує введення необхідних даних.
4. Адмін вводить дані та натискає кнопку збереження.
5. Система перевіряє введені дані та зберігає їх у базу даних.
6. Система повідомляє адміну про успішне збереження даних.

# НАЗВА: Редагування запису в базі даних

# УЧАСНИКИ:

- Адміністратор 
- Користувачі

# ПЕРЕДУМОВИ:

- Користувачі мають доступ до системи 
- Адміністратор має права на редагування записів

# РЕЗУЛЬТАТ:

- Зміни до запису внесені до бази даних

# ВИКЛЮЧНІ СИТУАЦІЇ:

- Відсутність доступу до системи
- Некоректні дані для редагування

# ОСНОВНИЙ СЦЕНАРІЙ:

1. Користувач відправляє запит на редагування запису до системи.
2. Адміністратор перевіряє запит на наявність всіх необхідних даних.
3. Адміністратор редагує запис у базі даних.
4. Система повідомляє користувача про успішне внесення змін до запису.
5. Користувач може переглянути змінений запис в системі.

# УЧАСНИКИ:

- Cистема

# ПЕРЕДУМОВИ:

- Адміністратор має права на додавання даних до системи та отримав запит на додавання даних від користувача.

# РЕЗУЛЬТАТ:

- Дані користувача успішно додані до системи.

# ВИКЛЮЧНІ СИТУАЦІЇ:

- Невірні дані в запиті
- Дублювання даних в системі

# ОСНОВНИЙ СЦЕНАРІЙ:

1. Адміністратор отримує запит на додавання даних від користувача.
2. Адміністратор перевіряє дані в запиті та переконується у їх правильності.
3. Адміністратор додає дані в систему.
4. Система перевіряє наявність дублюючих даних в системі.
5. Якщо дублюючих даних не знайдено, система підтверджує додавання даних в систему та повідомляє адміністратора та користувача про успішне додавання.
6. Якщо дублюючі дані знайдені, система повідомляє адміністратора та користувача про помилку та просить виправити дані в запиті.

5. Якщо дані введені правильно та в системі вже є дані користувача, система підтверджує внесення змін та повідомляє адміністратора та користувача про успішне редагування даних.
6. Якщо в системі відсутні дані користувача або дані введені неправильно, система повідомляє адміністратора та користувача про помилку та просить виправити дані в запиті.

# УЧАСНИКИ:

- Користувач має право на додавання даних в систему та має доступ до необхідного інтерфейсу.

# РЕЗУЛЬТАТ:

- Дані користувача успішно додані в систему та очікують на обробку адміністратором.

# ВИКЛЮЧНІ СИТУАЦІЇ:

- Невірні дані в запиті
- Перевищення обмежень на об'єм даних в системі
- Неочікувана помилка в процесі додавання даних

# ОСНОВНИЙ СЦЕНАРІЙ:

1. Користувач входить до системи та відкриває інтерфейс додавання даних.
2. Користувач вводить необхідні дані та підтверджує додавання.
3. Система перевіряє правильність введення даних та забезпечує їх безпеку перед збереженням.
4. Система зберігає дані у відповідну базу даних та повідомляє користувача про успішне додавання даних.
5. Адміністратор отримує повідомлення про новий запит на додавання даних та вирішує, чи підтверджувати додавання даних до системи.
6. Якщо запит на додавання даних підтверджено адміністратором, система зберігає дані відповідно до правил та обмежень, що діють в системі.
7. Якщо запит на додавання даних відхилено адміністратором, система повідомляє користувача про відхилення та причини його відхилення.

# УЧАСНИКИ:

- Користувач
- Система

# ПЕРЕДУМОВИ:

- Користувач має доступ до системи та знає ключові слова для пошуку даних

# РЕЗУЛЬТАТ:

- Користувач отримує список даних, які відповідають його запиту

# ВИКЛЮЧНІ СИТУАЦІЇ:

- Якщо користувач ввів невірні дані для пошуку, система повідомляє про помилку та просить ввести коректні дані.
- Якщо у системі немає даних, які відповідають запиту, користувач отримує повідомлення про те, що нічого не знайдено.

# ОСНОВНИЙ СЦЕНАРІЙ:

1. Користувач вводить ключові слова для пошуку даних.
2. Система перевіряє правильність введених даних та запускає процес пошуку.
3. Система повертає список даних, які відповідають запиту користувача.
4. Користувач переглядає результати пошуку.
5. Якщо користувач знаходить потрібні дані, то він може зберегти їх або перейти до редагування.
6. Якщо користувач не знаходить потрібні дані, то він може відредагувати запит та повторити пошук або створити новий запис.

# ID: user_reg

# НАЗВА: Реєстрація користувача

# УЧАСНИКИ:

- Користувач 
- Система

# ПЕРЕДУМОВИ:

- Користувач має доступ до Інтернету та відкритої сторінки реєстрації на сайті.

# РЕЗУЛЬТАТ:

- Користувач успішно зареєстрований в системі та може входити до свого облікового запису.

# ВИКЛЮЧНІ СИТУАЦІЇ:

- Користувач ввів некоректну інформацію в поле реєстрації.
- Користувач з таким іменем користувача вже існує в системі.

# ОСНОВНИЙ СЦЕНАРІЙ:

1. Користувач переходить на сторінку реєстрації на сайті та заповнює всі необхідні поля: ім'я, прізвище, адреса електронної пошти, ім'я користувача, пароль.
2. Система перевіряє правильність введеної інформації, перевіряє наявність введеного імені користувача в базі даних та повідомляє користувача про будь-які помилки.
3. Якщо інформація введена коректно, система створює новий обліковий запис користувача та зберігає його дані в базі даних.
4. Система автоматично перенаправляє користувача на сторінку входу до облікового запису.
5. Користувач вводить своє ім'я користувача та пароль, система перевіряє правильність введених даних та авторизує користувача.
6. Користувач отримує доступ до свого облікового запису та може користуватись системою.

# Функціональність

Функціональність керування даними залежить від кількох властивостей, які, у свою чергу, залежать від інтересів користувачів, які ними користуються. Однак існують різні рівні доступу до інформації, які можна поділити на: адміністратори та користувачі.

Дані кожного користувача є безкоштовними, доступними та придатними для повторного використання. Тому звичайні користувачі мають право вільно відтворювати, вимагати публікації, редагувати та використовувати дані за своїм бажанням.

Адміністратори також мають інші можливості, такі як: управління доступом до даних, видалення інформації або певних фрагментів, створення зручних інтерфейсів.

# Практичність

Контроль даних є найважливішим першим кроком ефективного широкомасштабного аналізу даних, який може дати значущу інформацію. Завдяки ефективному управлінню даними користувачі можуть знаходити та отримувати достовірні дані для своїх запитів.

# Надійність

Для підтримки надійності проєкту відкритої бази даних можна застосувати наступні підходи:

  1. Резервне копіювання даних: важливо регулярно створювати резервні копії бази даних, щоб у випадку втрати даних через збій апаратного забезпечення, кібератаку або інші подібні проблеми, можна було відновити дані з резервних копій.

  2. Захист від кібератак: важливо застосовувати заходи для захисту бази даних від кібератак, такі як встановлення надійного пароля, шифрування даних, застосування мережевих заходів захисту, таких як файрвол та інші.

  3. Моніторинг системи: важливо регулярно моніторити систему бази даних, щоб вчасно виявляти проблеми та усувати їх перед тим, як вони стануть серйозними.

  4. Постійні оновлення: важливо регулярно оновлювати програмне забезпечення, що використовується для роботи з базою даних, включаючи операційну систему, СУБД та інші програми.

  5. Навчання користувачів: важливо навчати користувачів проєкту відкритої бази даних засобам безпеки та практикам безпеки, таким як використання надійних паролів та уникаємо небезпечних дій в мережі Інтернет.

Застосування цих підходів допоможе забезпечити надійність проєкту відкритої бази даних і зменшити ймовірність втрати даних або їх несанкціонованого доступу.

# Продуктивність

  1. Швидкість реакції системи повинна бути достатньою, щоб користувачі не чекали надто довго на відповідь системи.

  2. Система повинна бути масштабованою, щоб забезпечити рост витрат на ресурси в разі збільшення обсягу даних та користувачів.

  3. Продуктивність системи повинна бути стабільною та надійною, щоб уникнути збоїв та забезпечити постійну доступність даних для користувачів.

  4. Важливо забезпечити належний рівень безпеки даних, щоб запобігти несанкціонованому доступу до них.

  5. Система повинна бути легкою в управлінні та підтримці, щоб забезпечити низькі витрати на обслуговування та підтримку.

  6. Для забезпечення швидкості та продуктивності системи важливо оптимізувати запити до бази даних та використовувати кешування даних.

  7. Для забезпечення масштабованості системи важливо використовувати віртуалізацію та контейнеризацію, щоб забезпечити гнучкість та розширюваність інфраструктури.

# Експлуатаційна придатність

Управління даними дає змогу ефективно масштабувати дані та використання за допомогою повторюваних процесів, щоб підтримувати дані та метадані в актуальному стані.
